# План проекта SmartTherm-помощника

## Вводные

### Цель проекта:

Создать технического помощника в виде Telegram-бота, который отвечает на вопросы по SmartTherm с привязкой к источникам (документация, репо, история telegram чата). Проект является курсовым проектом в университете. 

### Данные:

* История Telegram-чата, data/chat_history.json. Telegram чат является основным источником информации, он содержит вопросы пользователей, ответы разработчика, самостоятельные сообщения разработчика и пользователей.
* Официальная инструкция `instruction04.pdf` (техническая документация, схемы, распиновки).
* Репозиторий SmartTherm [https://github.com/Evgen2/SmartTherm/], локально: `data/SmartTherm`.
* Создан предварительный отчёт по проекту с планом дальнейших действий (`docs/initial_report.pdf`).

### Вычислительные ресурсы

* apple silicon m2 max 32gb
* kaggle GPU

## 1. Глубокий анализ вводных данных

### 1.2 Анализ Telegram-чата

**Статистика:**
- Всего сообщений: 36,133 (34,907 обычных + 1,226 сервисных)
- Разработчик: Evgen (7,114 сообщений)
- Период: ноябрь 2022 — март 2026 (~3.5 года)
- Типы контента: вопросы пользователей, ответы разработчика, объявления о версиях, ссылки на ресурсы, скриншоты ошибок

**Категории вопросов (выявленные паттерны):**
1. **Подключение оборудования** — распиновка, совместимость датчиков, подключение к котлу
2. **Прошивка и обновление** — процедуры прошивки, драйверы USB-COM, восстановление
3. **Диагностика ошибок** — коды ошибок OpenTherm, интерпретация статусов котла
4. **Настройка WiFi** — подключение к роутеру, точки доступа, IP-адреса
5. **Интеграция с умным домом** — Home Assistant, MQTT, TCP/UDP kgAPI
6. **Совместимость котлов** — поддержка различных моделей (BAXI, Navien, Immergas)
7. **PID-регулирование** — настройка температурных режимов, погодозависимая автоматика

### 1.3 Выводы по анализу данных

**Ключевые наблюдения:**
1. Чат содержит **непарные диалоги** — часто вопрос не имеет явного ответа, или ответ дан в нескольких сообщениях
2. Разработчик (Evgen) активно отвечает, но стиль ответов **краткий, технический, с предположением контекста**
3. Много **повторяющихся вопросов** — типичные проблемы подключения, прошивки, совместимости
4. Есть **объявления о версиях** — важны для отслеживания эволюции функционала
5. Присутствуют **ссылки на внешние ресурсы** — инструкции котлов, GitHub-репозитории, YouTube
6. В чате есть самостоятельные сообщения пользователей с их опытом решения проблем

Помощник не должен копировать стиль ответов разрабочика. сообщения разработчика часто они слишком сжатые или многочисленные в одной ветке. Также есть много самостоятельных сообщений пользователей, их обязательно использовать. Не нужно концентрироваться на разработчике, использовать для RAG и LoRA нужно все осмысленные сообщения.

## 2. План реализации (фазы)

### 2.0: Подготовка данных
1. **Обработка истории Telegram-чата** — Разделить чат на файлы по 3–4K сообщений. LLM анализирует каждую ветку (группы связанных сообщений) и генерирует два выходных формата: (а) для RAG — самодостаточные чанки cо связным текстом и описанием с метаданными для лучшего поиска. Текст агрегирует проблему, решения разработчика и опыт пользователей; (б) для LoRA — instruction-диалоги `{user, assistant}`, где ответ модели формируется на основе всей доступной информации в ветке. Обрабатываются все осмысленные сообщения: явные Q→A пары, самостоятельные сообщения пользователей о проблемах и решениях, объявления о версиях.
2. **Обработка инструкции (PDF)** — LLM генерирует Q-A пары на основе технических разделов (схемы, распиновка, процедуры подключения, коды ошибок). Для LoRA: синтетические диалоги `{user, assistant}` на основе типовых вопросов по документации.
3. **Обработка репозитория** — LLM анализирует ключевые файлы (прошивка, конфигурации) и создаёт чанки с контекстом: `{topic, code_snippet, description, file_path, lines}`. Только для RAG (код не используется в LoRA).
4. **Обработка истории версий (CHANGES.txt)** — LLM извлекает изменения по версиям: `{version, feature, bugfix, file}`. Добавляется в метаданные чанков для фильтрации по версии контроллера.
5. **Чанкинг** — разбиение на фрагменты с перекрытием 10-15% для улучшения retrieval.

### 2.1: RAG-only MVP
1. **Embeddings** — `BAAI/bge-m3` (dense) + BM25 (sparse) для гибридного поиска.
2. **Vector DB** — FAISS (HNSW индекс).
3. **RAG-пайплайн** — кастомный (без LangChain):
   * query rewriting — переформулировка вопроса для улучшения поиска;
   * retrieval: гибридный поиск (dense + sparse) → top-20 чанков;
   * reranking: cross-encoder (`bge-reranker`) для переранжирования top-20 → top-5;
   * контекст: чанки + системный промпт + метаданные (версия, источник);
   * генерация: ответ с указанием источников + валидация (ответ основан на retrieved чанках).
4. **Бот** — aiogram + FastAPI(prod) + Docker.
5. **Тесты** — 20-30 вопросов из чата, оценка точности.

### 2.2: LoRA
1. **Модель** — `Llama3.1-8B-Instruct`(base) или `Vikhr-Llama3.1-8B-Instruct-R`.
2. **Дообучение** — PEFT/LoRA, `r=8`, `alpha=16`, 3-5 эпох.
3. **Валидация** — сравнение с RAG-only на типичных вопросах.
4. **Где запускать** – QLoRA требует библиотеку bitsandbytes, которая недоступна на apple silicon. На apple silion доступна только LoRA.

### 2.3: Продакшен
1. **Inference** — Ollama.
2. **Мониторинг** — логи, метрики, алерты.

## 3. Финальные требования

### 3.1 Что будет сделано 

* Подготовлен корпус знаний из инструкции, репозитория и чатов (очистка, нормализация, анонимизация).
* Построен индексируемый векторный каталог фрагментов документации.
* Реализован базовый RAG-пайплайн: retrieval → контекст → генерация ответов с указанием источников.
* Дообучен (адаптирован) LLM-адаптер (LoRA/PEFT) на специализированных диалогах и Q→A *(опционально, после валидации RAG-only)*.
* Разработан Telegram-бот для диалогового интерфейса.
* Проведены тесты качества (авто + человеческая оценка) и настроена работа на сервере.

### 3.2 Ожидаемые артефакты

* Индексированная база знаний (векторный индекс + метаданные источников).
* Набор очищенных и размеченных обучающих примеров (JSONL).
* Сохранённый LoRA-адаптер (PEFT) и конфигурация инференса *(опционально)*.
* Рабочий Telegram-бот.

### 3.3 Технологии и библиотеки (ключевые)

* **Инструменты обработки данных:** Python, `pdftotext`/`pypdf`/`pdfminer`, OCR при необходимости.
* **Embeddings / retrieval:** `BAAI/bge-m3` (dense) + BM25 (sparse) → FAISS (HNSW). Гибридный поиск.
* **Reranking:** `bge-reranker` (cross-encoder) для переранжирования результатов.
* **LLM / дообучение:** Hugging Face `transformers` + `peft` (LoRA) + `bitsandbytes` + `accelerate`. Модель — `Vikhr-Llama3.1-8B-Instruct-R`.
* **RAG-оркестровка:** Кастомный пайплайн с query rewriting и response validation.
* **Сервис и интерфейс:** FastAPI (сервер), aiogram (бот), Docker.
* **Inference:** Ollama.
* **Мониторинг и логирование(возможно):** Prometheus + Grafana, ELK/Sentry для логов и ошибок.

### 3.4 Ключевые требования и риски

* **Фактическая достоверность:** ответы должны ссылаться на источник; RAG обязателен.
* **Воспроизводимость:** хранить версии базовой модели, адаптеров и индексов; Docker/requirements.
* **Ресурсы:** дообучение в режиме LoRA — экономно; инференс требует оценки latency/throughput.

### 3.5 Критерии успеха

| Метрика | Значение |
|---------|----------|
| Recall@5 (RAG) | > 0.85 |
| Фактическая точность | > 90% |
| Hallucination rate | < 5% |
